# Plugin Scaffolding

> Track: plugin-scaffold-20260428
> Type: chore

## Overview

Establish the foundational layout, manifest, and developer tooling for the `@pleaseai/spring` Claude Code plugin. The repository currently contains only documentation (`README.md`, `ARCHITECTURE.md`, `CLAUDE.md`, `LICENSE`) and the `.please/` workspace; there is no executable code, no plugin manifest, and no test/lint configuration. This chore creates the empty-but-correct skeleton so that subsequent feature tracks can land cleanly without bikeshedding tooling per PR.

This track introduces no business logic. Every artifact is a manifest, configuration file, or empty directory placeholder with the minimum content required to make the layout discoverable and the toolchain executable.

### Architectural direction (athens v2)

`ARCHITECTURE.md` describes a **static-install** pipeline that converts Antora HTML to Markdown at install time and ships prebuilt archives. The athens-v2 direction supersedes that pipeline:

- **Install-time output**: a thin skill manifest that records the resolved Spring component versions and points at a GitHub source ref. **No** Markdown is generated, **no** `prebuilt/` archives are produced or downloaded.
- **Runtime loading**: when Claude Code invokes the skill, it delegates to the `@pleaseai/ask` plugin (`ask src` / `ask docs`) to fetch the relevant Spring reference content from the upstream GitHub repository on demand.
- **Implication for this scaffolding track**: drop any layout that only existed to support the static pipeline (`prebuilt/`, Antora-conversion library scaffolding, nightly archive build workflow). Keep the standard plugin layout, runtime, lint, test, and CI tooling ‚Äî those are needed regardless of install strategy.

`ARCHITECTURE.md` will be revised in a separate track once the v2 direction is exercised end-to-end. This scaffolding track should not edit `ARCHITECTURE.md`.

## Scope

### Plugin layout (Claude Code conventions)

- `.claude-plugin/plugin.json` ‚Äî the **only** file in `.claude-plugin/`. Declares plugin name (`@pleaseai/spring`), version (`0.0.0`), description, license, and (if supported by the manifest schema) a declared dependency on `@pleaseai/ask`.
- `commands/` ‚Äî empty placeholder directory (with `.gitkeep`) for future `commands/*.md` slash command entry points.
- `skills/` ‚Äî empty placeholder directory for future `skills/spring-installer/SKILL.md` and the dynamically-loading per-component skills.
- `scripts/` ‚Äî empty placeholder directory for future detect/resolve/install orchestration scripts.
- `scripts/lib/` ‚Äî empty placeholder directory for future I/O-free helpers (build-file parsing, manifest schemas).

### Runtime & language tooling

- `package.json` ‚Äî declares Bun as the runtime, ESM modules, scripts (`test`, `lint`, `format`, `typecheck`), keeps direct deps minimal per the dependency policy in `tech-stack.md`.
- `tsconfig.json` ‚Äî TypeScript strict mode (`strict: true`, `noUncheckedIndexedAccess: true`, `noImplicitReturns: true`), ESNext target, `moduleResolution: "bundler"` to match Bun.
- `bun.lockb` ‚Äî generated by `bun install`; committed.

### Test / lint / format

- `bun test` works out of the box. A single placeholder test (`scripts/lib/__tests__/scaffold.test.ts`) asserts `1 + 1 === 2` to prove the runner is wired.
- ESLint flat config (`eslint.config.js`) targeting TypeScript with `@typescript-eslint`, plus a small set of hygiene rules. Formatting rules deferred to Prettier.
- Prettier config (`.prettierrc`, `.prettierignore`) ‚Äî sensible defaults; documented in `package.json` script (`bun run format`).

### Pre-commit & commit hygiene

- Husky (`.husky/pre-commit`) runs `lint-staged`.
- `lint-staged` config in `package.json` runs ESLint + Prettier on staged `.ts` files.
- `commitlint` is **out of scope** for this track (revisit if/when commit-message noise warrants).

### CI placeholder

- `.github/workflows/ci.yml` ‚Äî runs `bun install --frozen-lockfile`, `bun run typecheck`, `bun run lint`, `bun test` on PRs to `main`. Single Bun version, no matrix complexity.

### Documentation updates

- `README.md` ‚Äî append a short "Local Development" / "Project Layout" subsection if not already present. Do **not** rewrite or restructure.
- `ARCHITECTURE.md` ‚Äî **no changes**. Architectural pivot to v2 is documented separately.

## Constraints

- **No native dependencies** (`tech-stack.md` policy). All chosen tools are pure JS/TS.
- **Direct dependencies must be pinned** in `package.json` per dependency policy.
- **Scaffolding only**: zero `.ts` source files under `scripts/` or `skills/` other than the placeholder test. Any temptation to "while I'm here, add detect.ts" is deferred to the build-file detection track.
- **No bundler**: per `tech-stack.md`, Bun executes `.ts` directly. Do not introduce `tsup`, `esbuild`, `vite`, etc.
- **No `prebuilt/` directory or related tooling**: the v2 architecture does not ship prebuilt archives. Any layout justified solely by the static pipeline must be omitted.
- **Do not touch `ARCHITECTURE.md`**: the v2 pivot has its own track.
- **Do not modify the `<!-- spring-skill:start -->` markers convention**: that contract belongs to the install/annotate tracks; this scaffolding does not touch user-facing CLAUDE.md handling.

## Out of Scope

- Any `.ts` source under `scripts/` or `skills/` beyond a single placeholder test.
- Slash command implementations (`/spring:install`, `/spring:list`, etc.).
- Build-file parsing (Gradle/Maven), BOM resolution, version detection.
- Antora HTML ‚Üí Markdown conversion library and rules (no longer applicable under v2).
- `prebuilt/catalog.json` schema and content (no longer applicable under v2).
- Nightly archive build workflow (no longer applicable under v2).
- `@pleaseai/ask` integration code (separate feature track).
- Eval suite (`evals/spring/`) ‚Äî separate track.
- Release automation (`release-please`, semantic-release) ‚Äî separate track.
- `commitlint` configuration ‚Äî defer until commit-message noise warrants.
- `ARCHITECTURE.md` revision documenting the v2 pivot ‚Äî separate track.
- README rewrite or major restructure.
